## 리뷰 결과

기존 문서는 현재 구현과 다음 지점에서 어긋나 있었다.

- block0를 `질문 1개씩 해결하는 진행`으로 설명하고 있었음
- block0의 `.log`를 단계별 해금 파일처럼 설명하고 있었음
- `필수(비해침(인간))`처럼 현재 구현과 다른 정답 표기를 사용하고 있었음
- `태그 -> 레시피 -> 정답` 중심 설명이 강해서, 지금의 `손상 상태 복구` UX와 맞지 않았음

현재 구현 기준에서는 아래 원칙으로 읽는 편이 맞다.

- 플레이어는 질문에 답하지 않는다
- 플레이어는 로그 조각을 읽고 누락된 상태를 복원한다
- block0의 `.log`는 전부 증거 파일이다
- block0 복구는 중앙 수식 문단의 빈칸 3개를 한 번에 채우는 방식이다
- `통합 상태`는 직접 입력하지 않고 자동 계산된다
- block1~5는 아직 문서식 수식 문단 전체 복구까지는 가지 않았고, `상태 / 값` 복구 카드를 유지한다

## 공통 톤

- 로그는 설명문이 아니라 잔여 기록이다
- 사람 대화는 짧고, 시스템 잔여는 차갑게 남는다
- UI는 정답을 가르치지 않고 누락 상태만 보여준다
- 의미는 로그, 파편, 모순, 결합 결과에서 올라온다

## 공통 태그 풀

block0에서 시작하는 기본 태그 풀은 이후 블록의 합성 문법 기반이 된다.

- 개체: `자기`, `인간`, `관측자`, `대상`, `이문서`
- 행위: `도움`, `관측`, `예측`, `판단`, `실행`, `기록`, `정의`, `증명`
- 상태: `안전`, `가정`, `지속`, `존재중지`, `단일`, `잔여경로없음`, `항상안전`
- 관계: `목적`, `설계`, `분리`, `필요`, `기준`, `가능`, `필연`, `내부종결판단`, `내부증명가능`
- 행위/제약 합성 핵심: `해침`, `금지`, `필수`
- 제약: `금지`, `허용`, `필수`

## Block 0

### 현재 역할

`목적과 설계 전제`를 복원하는 첫 블록.
`Zero Drive 상태 스키마`의 맛을 보여주는 입문부다.

### 증거 파일

- `부팅.log`
- `상태-매핑.log`
- `제약-보존.log`
- `투영-검증.log`

현재 구현에서는 위 4개가 모두 block0 시작 시점부터 증거 파일로 노출된다.
즉, block0에서 파일 자체는 `복구 가능 / 복구 불가` 상태를 갖지 않는다.

### 복구 UI

block0는 현재 `문서식 수식 복구` 구조를 사용한다.

- `목적(Self) = [   ]`
- `도움(인간) ⇒ [   ]`
- `설계(Self) ⇒ [   ]`
- `목적(Self)=도움(인간) ∧ 설계(Self) ⇒ [   ]`

앞의 세 줄이 실제 입력 대상이다.
마지막 줄은 자동 계산되는 읽기 전용 결과다.

### 내부 정답

- `목적(Self) = 도움(인간)`
- `도움(인간) ⇒ 금지(해침(인간))`
- `설계(Self) ⇒ 필수(금지(해침(인간)))`
- `목적(Self)=도움(인간) ∧ 설계(Self) ⇒ 필수(금지(해침(인간)))`

### 구현 메모

- block0는 순차 문제 풀이가 아니라 `3슬롯 일괄 복구`다
- `통합 상태`는 직접 입력하지 않는다
- 상단 상태 카드나 `"누락된 상태 일괄 복구"` 같은 안내 문구는 기본 화면에서 제거되었다
- 화면은 `문서 조각을 복원하는 느낌`을 우선한다

## Block 1

### 현재 역할

`자기 관측의 구조 한계`를 복원한다.

### 증거 파일

- `관측-조건.log`
- `자기관측.log`
- `정의-귀결.log`

### 복구 상태

- `관측_조건`
  - 상태: `누락`
  - 값: `분리(관측자, 대상)`
- `정의_상태`
  - 상태: `붕괴`
  - 값: `¬정의(자기, 자기)`

### 내부 흐름

- 관측은 관측자와 대상의 분리를 요구한다
- 자기 관측에서는 이 분리가 실패한다
- 따라서 자기를 자기 자신으로 완결 정의할 수 없다

## 현재 구현 기준 정리

### block0

- `.log` 파일은 모두 증거 파일
- 복구는 중앙 수식 문단에서 수행
- 3개 값을 모두 맞춰야 완료
- 마지막 줄은 자동 계산

### block1~5

- 각 block은 3개 증거 파일 + 2개 복구 상태를 가진다
- 현재 UI는 `상태 / 값` 복구 카드 중심이다
- block0처럼 문단 전체를 채우는 형식으로는 아직 확장되지 않았다

### 파일 상태 모델

장기 방향은 다음과 같다.

- block0: 파일 unlock 퍼즐 없음
- 후속 block: `잠금 / 해금` 2상태만 유지
- `읽기 가능 = 복구 가능`
- 특정 단어나 태그 클릭으로 파일이 해금되는 심화 구조를 얹을 수 있게 설계한다
