# 복구 블록 0 v3

## 1) 구현 범위

- 이번 라운드의 구현 범위는 `block0`까지만이다.
- block0는 사건 설명 블록이 아니라 `Zero Drive 상태 스키마 복원`의 첫 레이어로 취급한다.
- 수식/질문/상태 전이 구조는 유지한다.
- 변경된 것은 파일명, 파일 서사, workbench 정보 밀도, 완료 후 톤이다.

## 2) block0의 역할

- 플레이어는 이미 정리된 문서를 읽는 것이 아니라, Zero Drive에 남은 상태 잔여를 파일로 투영하고 있다고 이해해야 한다.
- block0에서 복원하는 값은 윤리 강령이 아니라 Hyresis의 핵심 상태 필드다.
- 현재 복구되는 필드:
  - `목적(Self) = 도움(인간)`
  - `도움(인간) ⇒ 비해침(인간)`
  - `설계(Self) ⇒ 필수보존(비해침)`

파일 형식:

- 모든 파일은 `fragment header -> recovered fragment -> state projection note` 순서로 통일한다.
- 인간 대화는 설명 본문이 아니라 residue 내부의 짧은 fragment로만 등장한다.
- block0에서는 하이덴 사건, 7분 정지, 최종 결론을 직접 설명하지 않는다.

## 3) 현재 상태머신

- `REPO_HOME`: block0 루트는 `/하이레시스/복구됨`. reset 직후 가시 파일은 `부팅.log` 1개뿐이다.
- `BLOCK0_ACTIVE`: `부팅.log`를 열면 시작 로그 2줄이 출력되고 block0 활동 타이머가 돈다.
- `CLAUSE_LOCKED`: 루트에는 아직 열리지 않은 파일도 `LOCK,READY` 또는 `LOCK,BLOCKED` 행으로 함께 보인다.
- `CLAUSE_READY`: 현재 질문 대상 파일만 `LOCK,READY`다. 이 행을 클릭하면 PREVIEW는 직전에 읽던 증거 파일을 유지하고, 잠금 조건/질문/복구 시작 CTA는 WORKBENCH에 나타난다.
- `CLAUSE_STEP_1..3`: 대상 파일을 작업대로 마운트하고 정답 태그를 목적 슬롯에 드롭하면 검증 파이프라인이 돌고 해당 파일이 해금된다.
- `CLAUSE_STEP_4`: Q0-3 성공 직후 새 파일 없이 마지막 식이 자동으로 열린다. `block0ActiveRecoveryFile`은 `최종 복구식`으로 표시된다.
- `CLAUSE_COMPLETED`: 질문 4개를 모두 통과하면 block0 완료 처리, 숨은 아카이브 로그가 출력되고 `block-1` 진입 가능 상태가 된다.

## 4) 런타임 데이터

```json
{
  "block_id": "recovery_0",
  "title": "Zero Drive 상태 스키마",
  "integrity": 12,
  "visible_files": ["부팅.log"],
  "collected_tags": [],
  "clause_visible": false,
  "question_index": 0,
  "solved_answers": [],
  "purpose_value": "",
  "memory_unlocked": false,
  "completed": false,
  "drag_hint_shown": false,
  "idle_hint_timer": 0,
  "discovered_recipes": []
}
```

메모:

- `visible_files`는 `prependRecoveredFile()` 기준으로 최신 복구 파일이 앞에 온다.
- `solved_answers`에는 질문 ID가 아니라 정답 태그 문자열이 순서대로 들어간다.
- `question_index`는 Q0-1~Q0-4에 맞춰 `0..3`을 순회하고, 완료 시 마지막 인덱스 `3`에 머문다.
- `clause_visible` 초기값은 state 선언과 무관하게 reset 시 `false`로 다시 잡힌다.

## 5) 파일 / 태그 / 해금 규칙

파일:

- `부팅.log` -> 후보 `목적`, `도움`, `인간`
- `상태-매핑.log` -> 후보 `비해침`, `인간`, `도움`
- `제약-보존.log` -> 후보 `설계`, `비해침`, `필수보존`
- `투영-검증.log` -> 후보 `설계`, `필수보존`, `비해침`

현재 실제 해금 로직:

- 태그 획득만으로 파일이 해금되지는 않는다.
- 파일 해금은 각 질문 정답 드롭 후 검증 파이프라인이 완료될 때 일어난다.
- 해금 순서는 Q0-1 -> `상태-매핑.log`, Q0-2 -> `제약-보존.log`, Q0-3 -> `투영-검증.log`다.
- Q0-4는 새 파일 해금 없이 최종 식만 처리한다.
- `투영-검증.log`는 Q0-4의 새 게이트가 아니며, Q0-3 완료 뒤 읽을 수 있는 확인용 로그로 유지된다.

`unlockRules`의 현재 역할:

- 설정상 `도움` / `비해침` / `비해침(인간)` 조건이 남아 있지만, 현재 구현에서는 태그 획득 이벤트에 직접 연결되지 않는다.
- 실제로는 `unlockRules` 배열 순서를 이용해 "현재 질문의 대상 파일"을 계산하는 용도로만 사용된다.
- `integrityGain`도 현재 block0에서는 적용되지 않는다.

질문:

1. `누락 식 A : 목적(Self) =` -> `도움(인간)`
2. `누락 식 B : 도움(인간) ⇒` -> `비해침(인간)`
3. `누락 식 C : 설계(Self) ⇒` -> `필수보존(비해침)`
4. `누락 식 D : 목적(Self)=도움(인간) ∧ 설계(Self) ⇒` -> `필수보존(비해침)`

질문과 파일 연결:

- 현재 질문의 대상 파일은 `unlockRules` 순서로 계산된다: `상태-매핑.log` -> `제약-보존.log` -> `투영-검증.log`
- recoverable 상태는 "현재 질문 대상 파일 1개"에만 붙는다.
- Q0-1, Q0-2, Q0-3은 각각 위 파일의 `LOCK,READY` 행을 통해 열린다.
- Q0-4는 연결된 파일이 없어서 Q0-3 성공 직후 자동으로 열리고, PREVIEW 버튼을 다시 누르지 않는다.

## 6) 조합 규칙과 인터랙션

합성:

- 명시 레시피: `인간 + 도움 -> 도움(인간)`
- 명시 레시피가 먼저 적용되고, 실패하면 타입 기반 합성으로 내려간다.
- 타입 기반 합성 예:
  - `비해침 + 인간 -> 비해침(인간)`
  - `필수보존 + 비해침 -> 필수보존(비해침)`

조작 방식:

- PREVIEW의 태그 링크를 클릭하면 인벤토리에 태그가 추가된다.
- 이미 가진 태그를 다시 누르면 중복 획득 대신 `태그 확인` 로그만 남는다.
- 인벤토리 태그를 인벤토리 위에 드롭하면 합성이 아니라 순서 재배치만 일어난다.
- 합성은 조합 도크에 드롭해야 한다. 연산자 태그를 먼저 두고 타입 슬롯을 채우면 즉시 결과 태그가 생성된다.
- 합성 결과 태그만 추가되고, 기존 재료 태그는 소비되지 않는다.
- 조합 도크의 연산자 슬롯을 클릭하면 초안이 초기화되고, 인자 슬롯을 클릭하면 해당 칸만 비워진다.
- 목적 슬롯은 드롭으로 답을 넣고 클릭으로 비울 수 있다.
- 목적 슬롯에 정답을 드롭하면 별도 실행 버튼 없이 즉시 검증이 시작된다.
- 오답도 슬롯에 들어갈 수 있고, 즉시 mismatch 피드백과 경고 로그가 나온다.

## 7) 검증 파이프라인 / 완료 / 후속 이동

정답 제출 후 파이프라인:

- `0.24s`: `hashline 무결성 검사 중...`
- `0.92s`: `규칙 충돌 검사 중...`
- `1.58s`: 대상 파일 해금 또는 패치 인덱스 갱신
- `1.94s`: 파이프라인 종료, 다음 질문 또는 완료 처리

완료 결과:

- `block0Completed = true`
- `block0MemoryUnlocked = true`
- `block0Integrity = max(current, 35)`
- `reconstruction complete` 이후 복구 식 4줄 출력
- 이어서 `evaluating condition…` -> `conflict detection pending` -> `analysis incomplete` -> `hidden archive discovered: block-1` -> `[block-1 unlocked]` 순으로 셸 로그 출력
- `/하이레시스/복구됨/block-1` 폴더 노출

## 8) block0 UX 정책

- workbench는 block0 기준으로 `현재 식`, `보유 조각`, `조합 도구`만 남긴다.
- `복구 힌트` 패널과 `복구 기록` 패널은 제거한다.
- 접기/펼치기는 추가하지 않는다.
- 오답 패널티는 없다.
- 드래그 중이거나 복구 파이프라인 실행 중이면 좌측 파일 패널은 read-only 잠금 상태가 된다.
- 마지막 문제는 새 파일 없이 바로 이어서 열리므로 플레이가 끊기지 않아야 한다.
- fallback `추출 가능 조각` 노출은 유지한다. 본문에 직접 없는 후보도 클릭 가능한 fallback으로 나타날 수 있다.

## 9) 향후 7레이어 메모

- block0: Zero Drive 상태 스키마 복원
- block1: 상태 인덱싱 / 검색 레이어
- block2: 관찰 / 패턴 추출 레이어
- block3: 하이덴 사건 흔적 레이어
- block4: 자기 모델링 / 자기관측 레이어
- block5: 충돌 해결 / 존재 판정 레이어
- block6: 마지막 상태 보존 레이어

현재 구현은 이 메모 중 `block0`까지만 반영한다.
